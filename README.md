# mini2 — Multi-process distributed query overlay (CMPE 275)

A 9-node tree overlay built on **plain unary gRPC** that delivers a NYC 311
query result back to a client in **chunks the client paces itself**. The
spec forbids gRPC streaming and async APIs; we work around that by
implementing the streaming semantics in the application layer (see
`docs/ARCHITECTURE.md`).

This builds on the same NYC 311 dataset used in mini1.

## What's in the box

```
mini2-claude/
├── proto/overlay.proto         # Portal + Overlay services
├── config/topology.conf        # the entire topology lives here, NOT in code
├── cpp/                        # C++17 node + client (gRPC + protobuf)
├── python/                     # Python node (cross-language interop)
├── scripts/                    # build / run / partition / smoke
├── data/                       # populated by partitioner
├── docs/                       # ARCHITECTURE, CHUNKING, RUNNING, REPORT,
│                               # DESIGN_RATIONALE, RELATED_WORK, measurements.json
└── tests/                      # tiny pytest unit tests for parsing layers
```

## Quick start (one machine, smoke test with synthetic data)

```bash
cd mini2-claude

# 1) Python deps for the Python node and the proto generator
python3 -m pip install -r python/requirements.txt

# 2) Generate Python proto stubs (C++ stubs are generated by CMake at build time)
scripts/gen_proto.sh

# 3) Build the C++ node + client (needs cmake + grpc + protobuf installed)
scripts/build_cpp.sh

# 4) Synthetic data partitioned across the 9 nodes
python3 scripts/generate_synthetic.py --rows 50000 --outdir data \
        --nodes A,B,C,D,E,F,G,H,I

# 5) Bring up all 9 servers in the background
scripts/run_all_local.sh

# 6) Run a few queries through the portal (node A)
scripts/run_client.sh --borough BROOKLYN --max 200
scripts/run_client.sh --from 20230101 --to 20231231
scripts/run_client.sh --geo 40.6,40.8,-74.0,-73.9

# 7) Tear it down
scripts/kill_all.sh
```

For real NYC 311 data, replace step 4 with:

```bash
python3 scripts/partition_data.py \
    --input data/raw/311_2024.csv data/raw/311_2023.csv \
    --outdir data --nodes A,B,C,D,E,F,G,H,I
```

## Running across two machines

The topology file is the only thing you edit. Open
`config/topology.conf`, change the `host=` of each node so half live on
host1 and half on host2, then start them with `scripts/run_node.sh <ID>`
on the appropriate machine. Nothing else needs to change — peers find
each other via the same config file.

## Languages

* **C++17** — node server + client. The "real" implementation. Reuses
  the typed `Record311` schema and SoA storage idea from mini1.
* **Python 3** — second implementation of the node server. Speaks the
  exact same proto, so a Python node can be peered with C++ nodes
  freely. Default config makes nodes G and I Python.

## Key design decisions (one-liners)

1. **No gRPC streaming or async APIs.** All RPCs are blocking unary.
   Streaming is built on top with `PushChunk` (results flow up the
   tree as repeated unary calls in the reverse of the fan-out
   direction) plus `FetchChunk` (client pulls).
2. **Client paces delivery.** A buffers per-request rows in a bounded
   queue. Slow clients fill the buffer → upstream `pushRows` returns a
   short-count → producers throttle. Fast clients drain quickly → A
   raises `suggested_next_max` via an EWMA on fetch cadence.
3. **No hard-coded ids/ports/peers.** A node is just `node_server
   --id X --config topology.conf`.
4. **Typed columns, not strings-for-everything.** `unique_key` is
   `uint64`, dates are packed `uint32` YYYYMMDD, borough is a 1-byte
   enum, lat/lon are `double`, and the proto mirrors that.
5. **No replication.** Each node owns its slice of the data
   (`hash(unique_key) % N`). Partition files live next to the node.
6. **Cancel + abandonment handling.** Client `CancelQuery` propagates
   downstream as `Abort`. A janitor thread garbage-collects requests
   whose last `FetchChunk` was longer than `request_ttl_secs` ago,
   covering disconnects and abandoned requests.

## Reproducing the measurements in the report

```bash
python3 scripts/generate_synthetic.py --rows 50000 --outdir data
# start the all-python cluster (or use scripts/run_all_local.sh for mixed)
for N in A B C D E F G H I; do
  python3 python/node_server.py --id $N --config config/topology_allpy.conf &
done
python3 scripts/measure.py --reps 3 --slow-ms 100 --out docs/measurements.json
```

`config/topology_allpy.conf` is the measurement variant that makes every
node Python, so the numbers are reproducible without needing gRPC C++
installed. `config/topology.conf` is the production variant with C++ for
7 nodes and Python for G, I.
